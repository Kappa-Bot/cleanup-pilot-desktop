# Release Operations

## Release model

Cleanup Pilot Desktop is packaged for Windows with `electron-builder` and an NSIS installer. Release artifacts are published to GitHub Releases, and the packaged app uses `electron-updater` to check the GitHub release feed.

The release pipeline is intentionally narrow:

- one installer target: Windows NSIS
- one update source: GitHub Releases
- one release trigger: semver tags that start with `v`

## Local packaging

To build a local installer without publishing:

```bash
npm ci
npm run build
npm run package:win
```

This creates the Windows installer and update metadata in `release/`.

## Publishing a release

1. Bump `package.json` version.
2. Merge the change to `main`.
3. Create and push a tag that matches the version, for example `v0.1.1`.
4. The GitHub Actions release workflow runs on the tag, validates the repo, builds the app, and publishes the NSIS installer to GitHub Releases.

The release job uses the repository `GITHUB_TOKEN` via `GH_TOKEN`. No publishing secret should be checked into the repo.

## Update checks in the app

The renderer still calls `desktopApi.checkUpdates()`, which continues to invoke the `updates.check` IPC channel.

Update resolution is:

1. Prefer GitHub Releases through `electron-updater` when the app is packaged.
2. Fall back to the legacy JSON feed only if packaging update metadata is unavailable and `updatesFeedUrl` is configured.
3. Return a no-update response if neither source is available.

## Verification checklist

- Confirm the tag name matches the app version prefix `v`.
- Confirm the GitHub release contains the `.exe` installer and the update metadata file generated by `electron-builder`.
- Confirm the installer starts and `Help -> Check updates` or the settings button still returns a valid `UpdateCheckResponse`.
- Confirm the app loads without navigation warnings in the packaged build.

## Rollback

If a bad build ships, delete or unpublish the GitHub Release and cut a replacement tag for the corrected version. The app checks GitHub Releases at runtime, so the next packaged launch will use the latest published release metadata.
